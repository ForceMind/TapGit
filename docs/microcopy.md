# 码迹（TapGit）文案规范

## 1. 总体语气
- 像带用户做事，不像给用户上课
- 先说“现在是什么情况”，再说“下一步点哪里”
- 默认不用 Git 原词
- 少解释原理，多解释动作

## 2. 首屏入口文案
- `打开本地项目`
- `从 GitHub 获取项目`
- `选择电脑里的项目文件夹`
- `输入仓库地址，拉到本地后自动打开`

不要写：
- `Open Repository`
- `Clone Repository`
- `Select Repo`

## 3. 顶部菜单文案
- `项目`
- `进度`
- `试新想法`
- `云端`
- `帮助`

二级菜单建议：
- `打开本地项目`
- `从 GitHub 获取项目`
- `查看当前修改`
- `保存全部修改`
- `查看保存记录`
- `新建想法副本`
- `切回稳定版本`
- `连接 GitHub`
- `上传到云端`
- `获取最新内容`
- `导出日志`

## 4. “试新想法”文案要求
始终围绕“安全试验”来表达。

推荐说法：
- `先开一个想法副本，再放心试`
- `你的稳定版本不会被直接改坏`
- `满意了，再把它带回当前版本`

不要写：
- `管理分支`
- `切换 branch`
- `merge branch`

## 5. 首页文案约束
- 标题尽量不超过 8 个汉字
- 描述尽量不超过 2 句
- 同一屏不要出现多个一模一样的按钮文案
- 当前最重要动作只保留一个主按钮

## 6. 获取项目相关文案
- 标题：`从 GitHub 获取项目`
- 仓库地址占位：`例如：https://github.com/owner/repo.git`
- 目标位置按钮：`选择保存位置`
- 开始按钮：`开始获取`
- 成功提示：`项目已获取并打开`
- 失败提示：`获取项目失败，请检查地址、网络或登录状态`
- 已存在目录提示：`这个位置已经有同名文件夹了，请换一个位置或名称`

## 7. 保存与恢复文案
- `保存进度`
- `全部保存`
- `只保存勾选内容`
- `恢复到这里`
- `回到这个安全备份`

恢复确认建议：
- 标题：`要恢复到这个保存点吗？`
- 正文：`当前项目文件会回到这个时间点，尚未保存的内容可能被替换。`
- 补充：`系统会先创建安全备份，方便你回到现在。`

## 8. 出错时的表达方式
推荐结构：
1. 这次哪里没成功
2. 当前内容有没有丢
3. 下一步怎么做

示例：
- `这次保存没有成功，你当前的修改还在。请重试。`
- `还没有连接云端，请先连接 GitHub 或填写仓库地址。`
- `这个项目还没开启版本保护，先开启后才能保存进度。`
