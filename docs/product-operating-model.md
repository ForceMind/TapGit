# 码迹操作流程模型

## 核心判断

码迹不应该像 Git GUI 一样把功能按命令分类，而应该按“这次修改处在哪个节点”来组织。

普通用户真正关心的是：

1. 我现在改了哪些文件。
2. 这个状态要不要保存成一个节点。
3. 我能不能从某个节点另开一条路线试新想法。
4. 试完之后，能不能把结果带回稳定路线。
5. 出错时，能不能回到某个节点。
6. 需要协作时，能不能把节点上传或获取下来。

所以产品主线应当是：

打开项目 -> 查看当前修改 -> 保存成节点 -> 围绕节点查看历史/恢复/试新想法/同步。

## 主流程

### 1. 打开项目

用户只有两个入口：

- 打开本地项目
- 从 GitHub 获取项目

打开后首页必须聚焦这个项目，不再堆叠大量引导文案。项目卡片只展示当前状态、最近保存、当前路线和快速操作。

### 2. 当前修改

当前修改页回答一个问题：这次修改是否要保存。

页面结构：

- 左侧：改动文件列表。
- 右侧：变化详情。
- 底部：保存说明和保存按钮。

可用操作：

- 保存全部修改。
- 选择部分文件保存。
- 在文件夹中删除文件。
- 停止追踪某个文件。
- 丢弃当前修改，且先保留安全备份。

### 3. 保存记录

保存记录页回答一个问题：我要找回哪个节点。

页面结构：

- 左侧：按时间排列的保存节点。
- 右侧：选中节点的文件变化。

可用操作：

- 浏览保存节点。
- 查看节点详情。
- 恢复到这里，恢复前自动创建安全备份。

### 4. 试新想法

试新想法本质是分支，但用户不需要先理解分支。

界面表达为：

- 稳定路线：用户可随时回来的主路线。
- 想法路线：从某个保存节点拉出的独立路线。
- 横向时间线：展示保存节点如何向前推进，以及想法路线从哪里分出来。

可用操作：

- 从当前节点创建想法路线。
- 切换到某条路线。
- 将想法路线带回稳定路线。
- 两条路线改到同一处时，用左右对比决定保留哪份。

### 5. 同步

同步页不再是参数表单，而是三步流程：

1. 连接账号。
2. 连接这个项目的云端地址。
3. 按保存节点上传或获取。

用户语言：

- 上传到云端。
- 获取云端最新内容。
- 当前有几个节点等待上传或获取。

## 保留、增加、隐藏

保留：

- 打开本地项目。
- 从 GitHub 获取项目。
- 保存当前修改。
- 查看保存记录。
- 恢复到保存节点。
- 创建想法路线。
- 切换想法路线。
- 带回稳定路线。
- 上传和获取云端内容。
- 账号登录和退出。

增加：

- 左下角账号入口可直接登录。
- 试新想法横向路线图。
- 同步三步式流程。
- 每个阻塞状态给出下一步按钮。

隐藏：

- AI 助手，直到真实可用。
- rebase、cherry-pick、stash、force push 等高级操作。
- 默认界面中的 Git 术语。

## 图形化原则

1. 所有核心操作围绕“保存节点”解释。
2. 页面只允许一个主任务，不在同一页混杂太多入口。
3. 分支用“路线”表达，提交用“保存节点”表达，远程用“云端”表达。
4. 遇到危险操作先告诉用户会保留备份，再让用户确认。
5. 如果用户下一步不可做，界面必须说明原因并给出可点击的下一步。
