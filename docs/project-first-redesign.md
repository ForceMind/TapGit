# 项目优先改版说明

## 目标

当用户已经打开一个项目后，界面不再继续强调“教学”和“全局入口”，而是只回答四个问题：

1. 这是哪个项目
2. 当前是不是安全状态
3. 现在最该做的一步是什么
4. 我去哪里处理修改、历史、试验

## 首页改成什么

打开项目后，首页只保留三层信息：

1. 项目概览
   - 项目名
   - 项目路径
   - 当前工作副本 / 未保存修改 / 保存记录数

2. 当前一步
   - 只保留一个主动作
   - 主动作根据项目状态自动变化
   - 优先级顺序固定：
     - 还没打开项目：打开项目
     - 还没开启版本保护：开启版本保护
     - 有未保存修改或还没有第一条保存：去保存
     - 已经稳定：去看历史

3. 三个工作区
   - 修改：查看当前改了什么，保存这次进度
   - 历史：查看之前保存过的版本，恢复到某个时间点
   - 试验区：在单独副本里试新想法，再决定是否带回稳定版本

## 明确不再做的事

- 不在已打开项目首页堆“最近项目”
- 不在首页摆一整套多步教学卡片
- 不在首页并列放 5 到 6 个同级按钮
- 不在顶栏长期显示云端等次要信息
- 不让“试验区”看起来像给 Git 老手用的分支管理页

## 导航原则

左侧导航统一压缩成短词：

- 项目
- 修改
- 历史
- 试验区
- 设置

目的不是少字，而是让用户一眼扫到“我现在要去哪里”。

## 顶部菜单原则

顶部菜单不再沿用系统默认的“文件 / 编辑 / 查看 / 窗口 / 帮助”。

改成任务菜单：

- 项目
- 修改
- 试验区
- 云端
- 更多

每个菜单只放真正有用的动作，不放重复入口和技术性操作。

## 评判标准

这版界面是否合格，不看信息量多少，只看三件事：

1. 用户打开项目后，是否能在 3 秒内看懂当前项目状态
2. 用户是否能在不读大段说明的前提下找到下一步
3. 用户是否会把“试验区”理解成一个安全实验区，而不是 Git 分支术语包装
