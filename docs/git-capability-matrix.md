# Git 能力矩阵

## 说明

这份矩阵回答一个问题：TapGit 现在到底覆盖了哪些常见 Git 动作，哪些还没有。

判断依据：

- 后端实现：[/E:/Privy/TapGit/electron/main/git-service.ts](/E:/Privy/TapGit/electron/main/git-service.ts)
- 桥接接口：[/E:/Privy/TapGit/src/shared/contracts.ts](/E:/Privy/TapGit/src/shared/contracts.ts)
- 当前页面入口：[/E:/Privy/TapGit/src/pages](/E:/Privy/TapGit/src/pages)

这是“能力审计”，不是逐项人工点击回归。它能准确说明“有没有做”，但不等于“所有支持项都已经做完完整手测”。

状态说明：

- `已支持`：产品里已有入口，后端也有实现
- `部分支持`：底层能力存在，但交互受限，或只支持部分场景
- `未支持`：当前没有 UI 和桥接能力

## 核心结论

TapGit 目前已经覆盖了这组高频动作：

- 获取项目
- 开启版本保护
- 查看修改
- 保存进度
- 查看历史
- 恢复到历史点
- 创建试验副本
- 切换试验副本
- 合并试验副本
- 处理合并冲突
- 连接云端
- 上传到云端
- 获取云端最新内容

TapGit 目前还没有覆盖这组常见 Git 动作：

- 贮藏修改
- 重置最近一次提交
- 撤销某次提交
- 整理历史
- 拣选提交
- 删除副本
- 重命名副本
- 标签管理
- 强制上传

## 能力表

| 用户任务 | 对应 Git 动作 | 当前状态 | 当前入口 | 说明 |
| --- | --- | --- | --- | --- |
| 从远端拿到一个项目 | `git clone` | 已支持 | 首页“从 GitHub 获取项目” | 当前只做了 GitHub 获取向导，本质是 clone |
| 打开本地项目 | 仓库检测 | 已支持 | 首页“打开本地项目” | 会检测是否已是 Git 仓库 |
| 开启版本保护 | `git init` | 已支持 | 首页 / 项目页“开启版本保护” | 不直接暴露 `git init` |
| 查看当前改了什么 | `git status` | 已支持 | 修改页 | 能看到文件级变更 |
| 查看差异内容 | `git diff` / `git diff --cached` | 已支持 | 修改页右侧预览 | 同时展示已暂存和未暂存差异 |
| 只保存部分文件 | `git add <files>` | 部分支持 | 修改页 | 只支持按文件选择，不支持按代码块选择 |
| 保存全部修改 | `git add -A` + `git commit` | 已支持 | 修改页 | 产品文案叫“保存进度” |
| 查看保存历史 | `git log` | 已支持 | 历史页 | 展示时间线和文件数 |
| 恢复到某个历史点 | `git reset --hard <commit>` | 已支持 | 历史页 | 恢复前可自动建安全备份 |
| 查看恢复前备份 | `git branch safety/...` | 已支持 | 历史页“安全备份” | 备份对用户可见 |
| 新建试验副本 | `git switch -c` | 已支持 | 试验区 | 对用户隐藏 branch 概念 |
| 切换试验副本 | `git switch` | 已支持 | 试验区 | 切换前未保存修改会失败并提示 |
| 查看所有副本 | `git for-each-ref refs/heads` | 已支持 | 试验区 | 当前隐藏技术分支，只展示用户关心的副本 |
| 把试验带回稳定版本 | `git merge --no-ff --no-edit` | 已支持 | 试验区 | 合并前可自动建安全备份 |
| 解决冲突 | `git checkout --ours/--theirs` / 手动编辑 + `git add` | 已支持 | 试验区冲突决策界面 | 当前是文件级决策，不是块级决策 |
| 完成冲突后的合并 | `git commit` | 已支持 | 试验区冲突决策界面 | 所有冲突处理后完成保存 |
| 连接云端 | `git remote add` / `git remote set-url` | 已支持 | 设置页 / 云端工作台 | 当前主要围绕单远端模型 |
| 测试云端地址 | `git ls-remote --heads` | 已支持 | 设置页 / 云端工作台 | 可区分权限、地址和网络问题 |
| 上传到云端 | `git push` / `git push -u` | 已支持 | 设置页 / 云端工作台 | 首次上传会自动设置跟踪关系 |
| 获取云端最新内容 | `git fetch` + `git pull --ff-only` | 已支持 | 设置页 / 云端工作台 | 要求本地没有未保存修改 |
| 只获取，不合并 | `git fetch` | 部分支持 | 无独立入口 | 当前只在“获取云端最新内容”内部使用 |
| 查看本地与云端差多少 | tracking 状态查询 | 已支持 | 顶栏 / 设置页 | 已显示待上传、待下载数量 |
| 登录 GitHub 账号 | Git Credential Manager | 已支持 | 设置页 | 这是同步辅助能力，不是 Git 原生命令 |

## 当前缺失的常见 Git 能力

| 常见动作 | 对应 Git 命令 | 当前状态 | 说明 |
| --- | --- | --- | --- |
| 临时收起当前修改 | `git stash` | 未支持 | 当前没有 stash 的 UI、桥接和后端实现 |
| 取回之前收起的修改 | `git stash pop` / `git stash apply` | 未支持 | 依赖 stash，当前一起缺失 |
| 查看 stash 列表 | `git stash list` | 未支持 | 当前没有相关能力 |
| 撤回最近一次保存 | `git reset --soft HEAD~1` / `git reset --mixed HEAD~1` | 未支持 | 当前只有“恢复到某个历史点”，而且是硬恢复 |
| 撤回最近几次保存 | `git reset` | 未支持 | 没有显式 reset UI |
| 反向撤销某次保存 | `git revert` | 未支持 | 当前没有更安全的“反向保存”流程 |
| 丢弃某个文件的未保存修改 | `git restore --source=...` / `git checkout --` | 未支持 | 当前没有单文件丢弃入口 |
| 删除试验副本 | `git branch -d` / `git branch -D` | 未支持 | 现在只能创建、切换、合并 |
| 重命名试验副本 | `git branch -m` | 未支持 | 当前没有重命名入口 |
| 修改最近一次保存说明 | `git commit --amend` | 未支持 | 当前保存后不可编辑说明 |
| 整理历史 | `git rebase` | 未支持 | MVP 明确不做 |
| 拣选某次保存 | `git cherry-pick` | 未支持 | MVP 明确不做 |
| 建立标签 | `git tag` | 未支持 | 当前没有版本标签管理 |
| 强制上传 | `git push --force` | 未支持 | 为了降低风险，当前刻意不做 |
| 多远端复杂管理 | 多个 `remote` | 未支持 | 当前按单远端模型设计 |
| 子模块 | `git submodule` | 未支持 | 当前不支持 |
| LFS | Git LFS | 未支持 | 当前不支持 |

## 产品层面的判断

如果站在“普通人保存代码进度”的目标看，当前最关键的主链路已经有了：

1. 打开或获取项目
2. 开启版本保护
3. 看懂改动
4. 保存进度
5. 回到历史点
6. 安全试新想法
7. 合并试验结果
8. 上传和获取云端内容

如果站在“常见 Git 能力覆盖”看，当前最值得补的缺口有 4 个：

1. 贮藏修改  
   这是切换副本和临时处理中断最常见的动作。
2. 撤回最近一次保存  
   普通用户会把它理解成“刚刚那次保存想撤销”。
3. 丢弃未保存修改  
   这是修改页的高频动作，目前缺失。
4. 删除试验副本  
   做完试验后，用户需要清理副本。

## 建议的下一轮测试顺序

1. 先把“已支持”的核心链路做成稳定的手测和自动化 smoke test
2. 再补“贮藏修改”“撤回最近一次保存”“丢弃未保存修改”“删除试验副本”
3. `rebase`、`cherry-pick`、`force push` 继续保持不做或仅高级模式隐藏
